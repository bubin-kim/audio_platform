# PRD — 오디오 데이터셋 관리 플랫폼 (Audio Dataset Management Platform)

> **문서 목적**: 이 문서는 "무엇을, 왜 만드는가"를 정의합니다. "어떻게 만드는가"(아키텍처·코드)는
> 이후 문서(02~)에서 다룹니다. 이 PRD는 Claude Code로 개발할 때 판단 기준이 되는 최상위 문서입니다.

- **버전**: v1.0 (MVP 기준)
- **작성일**: 2026-07-10
- **한 줄 정의**: 대학 연구실이 **여러 오디오 연구 프로젝트에 걸쳐 재사용**할 수 있는, 수집→전처리→메타데이터→데이터셋→연구기록 자동화 플랫폼.

---

## 1. 배경과 문제 정의 (Why)

연구실은 오디오 프로젝트마다 아래 작업을 **매번 수작업으로 반복**한다.

| 작업 | 현재 방식 | 문제 |
|---|---|---|
| 긴 녹음 커팅 | librosa 스크립트를 매번 새로 작성 | 프로젝트마다 코드 재작성, 재현 어려움 |
| 파일명 규칙 | 사람이 손으로 명명 | 규칙 불일치, 오타, 조건 누락 |
| 메타데이터 정리 | 엑셀 수기 입력 | 시간 소모, 실수, 버전 관리 안 됨 |
| 진행률 파악 | 폴더를 눈으로 셈 | "목표 대비 얼마나 모았나"를 알기 어려움 |
| 연구 기록 | 개인이 따로 정리 | 흩어짐, 인수인계 어려움 |

**핵심 문제**: 프로젝트가 바뀔 때마다(차량음 → 심음 → 산업음) 위 과정을 처음부터 다시 만든다.
도메인마다 파이프라인이 새로 만들어지므로 **재사용성이 0에 가깝다.**

## 2. 목표와 비목표 (Goals / Non-Goals)

### 2.1 목표 (Goals)
- **G1. 도메인 독립성**: 차량음·환경음·심음·음성·산업음·의료음 등 어떤 오디오 연구에도 코드 수정 없이 재사용.
- **G2. 반복 작업 자동화**: 커팅·메타데이터 추출·파일명 생성·CSV 생성을 자동화.
- **G3. 가시성**: 대시보드로 데이터 현황·진행률·통계를 한눈에.
- **G4. 유지보수 가능성**: 학부 연구생이 인수받아도 유지·확장할 수 있는 스택과 구조.
- **G5. 확장성**: MCP(Drive/Notion/GitHub)·AI Assistant·인증을 **나중에 플러그인처럼** 붙일 수 있는 골격.

### 2.2 비목표 (Non-Goals, MVP에서 안 함)
- 실시간 스트리밍 녹음(플랫폼은 "업로드된 파일"을 다룸).
- 오디오 라벨링용 정교한 파형 편집 UI(수동 미세 편집기).
- 사용자 로그인/권한(V2). MVP는 단일 연구실 로컬 사용 가정.
- 클라우드 배포·멀티테넌시. MVP는 로컬 실행.
- 딥러닝 모델 학습 기능(플랫폼은 "데이터셋 준비"까지 담당, 학습은 별개).

> **비목표를 명시하는 이유**: 범위가 무한히 커지는 것을 막기 위함. "안 하는 것"을 정하는 게 "하는 것"만큼 중요하다.

## 3. 사용자 (Who)

| 페르소나 | 역할 | 주요 사용 시나리오 | MVP 권한 |
|---|---|---|---|
| 연구원(대학원생) | 데이터 수집·전처리 | 녹음 업로드 → 커팅 → 데이터셋 만들기 | 전체 사용 |
| 학부 연구생 | 반복 작업 수행 | 정해진 프로젝트 설정으로 업로드·처리 | 전체 사용 |
| 교수 | 진행 상황 확인 | 대시보드로 각 프로젝트 진척 확인 | 전체 사용 |

> MVP에서는 권한 구분이 없다(모두 동일). V2에서 JWT로 교수/연구원/관리자를 분리한다.
> 지금 권한을 넣지 않는 이유: 인증은 기능 검증이 끝난 뒤 붙여도 늦지 않으며, 초반에 넣으면 학습·개발 속도를 크게 떨어뜨린다.

## 4. 핵심 개념 모델 (플랫폼을 재사용 가능하게 만드는 열쇠)

이 플랫폼의 재사용성은 **3개의 핵심 개념**에서 나온다.

### 4.1 Project (프로젝트)
하나의 연구 단위. **도메인의 차이는 전부 여기에 담긴다.**
- 커팅 전략 (event / silence / fixed_interval)과 파라미터
- 파일명 규칙 (naming pattern)
- 이 프로젝트가 요구하는 라벨 필드 (label schema) — 예: 차량음은 `distance/direction`, 심음은 `patient_id/valve`

> 예시: "차량음 프로젝트"와 "심음 프로젝트"는 **같은 코드**를 쓰지만
> Project 설정이 달라서 서로 다르게 동작한다. 이게 도메인 독립성의 실체다.

### 4.2 Dataset (데이터셋)
Project 안에서 만들어진, **버전이 있는 데이터 묶음**.
- 하나의 Project는 여러 Dataset을 가질 수 있다 (예: v1 초기수집, v2 재수집).
- Dataset은 여러 오디오 세그먼트 + 그 메타데이터로 구성된다.
- V2에서 GitHub로 버전 관리되는 단위가 이 Dataset이다.

### 4.3 Segment / Metadata (세그먼트와 메타데이터)
커팅으로 생성된 개별 오디오 조각 하나 = 한 레코드.
- **자동 추출 메타데이터**: duration, sample_rate, channel, bit_depth, file_size, created_time
- **라벨(사람/설정이 부여)**: Project의 label schema에 따라 달라지는 조건값

> 이 3층 구조(Project → Dataset → Segment) 덕분에, 어떤 오디오 도메인이 와도
> "새 Project를 만들고 그 안에 Dataset을 쌓는다"는 **동일한 흐름**으로 처리된다.

## 5. MVP 기능 요구사항 (What — 반드시 되는 것)

### F1. 업로드
- 단일 오디오 파일 또는 폴더(다중 파일) 업로드.
- 지원 포맷: wav 우선, mp3/flac/m4a(ffmpeg 경유).
- 업로드 이력(Upload History) 기록.

### F2. 메타데이터 자동 추출
업로드 시 각 파일에서 자동 추출: Duration, Sample Rate, Channel, Bit Depth, File Size, Created Time.

### F3. 자동 커팅
- 전략을 **Project 설정으로 선택**: Event Detection / Silence Detection / Fixed Interval.
- 긴 녹음(예: 4시간) → 여러 wav 세그먼트로 자동 분할.
- 처리 이력(Processing History) 기록.

### F4. 파일명 자동 생성
- **규칙을 사용자가 설정** (naming pattern).
- 예: `{date}_{model}_{distance}_{direction}_{seq:03d}.wav` → `20260710_EV6_10m_Front_001.wav`
- 규칙에 쓰인 필드는 Project의 label schema + 자동 메타데이터에서 채워진다.

### F5. Metadata.csv 자동 생성
- Dataset 단위로 세그먼트 메타데이터를 CSV로 내보내기.
- 한 행 = 한 세그먼트. 자동 메타데이터 + 라벨 컬럼 포함.

### F6. 대시보드
전체/프로젝트별로 다음 시각화:
- 총 데이터 개수, 총 녹음 시간, 저장 용량, 평균 길이
- 프로젝트별 데이터 현황
- 업로드 진행률, 라벨링 진행률
- Sample Rate 분포, 파일 형식 통계
- 최근 업로드 파일

## 6. V2 기능 (나중 — 지금은 "자리만" 만든다)

| 영역 | 기능 | MVP에서 준비할 것 |
|---|---|---|
| Google Drive (MCP) | wav/csv 자동 업로드, 프로젝트별 폴더 | Storage를 인터페이스로 추상화 (7.R2) |
| Notion (MCP) | 데이터셋 기록, 연구일지·회의록 자동화 | 이벤트 훅 지점 확보 (7.R5) |
| GitHub (MCP) | Dataset 버전 관리 | Dataset을 버전 단위로 모델링 (4.2) |
| AI Assistant | 자연어 질의("20m 데이터 몇 개?"), 라벨 추천, 요약 | API를 명확히 두어 나중에 도구로 노출 |
| Auth | JWT 기반 교수/연구원/관리자 권한 | 라우터 계층 분리로 후속 삽입 쉽게 |

> 원칙: **V2 기능은 지금 만들지 않는다. 그러나 "붙일 자리"는 지금 만든다.**
> MCP·AI·Auth가 없어도 플랫폼은 완전하게 동작해야 한다(플러그인 방식).

## 7. 비기능 요구사항 (How well — 품질 기준)

- **R1. 재사용성**: 새 오디오 도메인 추가 = 코드 변경 없이 Project 설정 추가만으로 가능해야 한다. *(최우선)*
- **R2. 확장성 — 저장소**: Local Storage와 향후 Drive를 **동일한 인터페이스**(StorageBackend)로 다뤄, 구현체 교체만으로 전환 가능.
- **R3. 확장성 — DB**: MVP는 SQLite, 향후 PostgreSQL. **ORM(SQLAlchemy)** 을 써서 SQL 종속을 피하고 마이그레이션(Alembic)으로 전환.
- **R4. 유지보수성**: 학부 연구생이 읽을 수 있는 구조. 계층 분리(라우터/서비스/처리 로직), 명확한 네이밍, 자동 API 문서(Swagger).
- **R5. 확장성 — 연동 훅**: 업로드·커팅·CSV 생성 같은 주요 사건에 **이벤트 훅** 지점을 두어, 나중에 Notion 기록·Drive 업로드를 훅으로 붙인다.
- **R6. 학습 곡선 배려**: 사용자는 React/FastAPI 초심자. 화려함보다 **표준적이고 문서 많은 패턴**을 우선. 남들이 다 쓰는 길로 간다.
- **R7. 안정성 우선**: 최신 기술보다 **성숙하고 문서화된** 도구. 실험적 라이브러리 지양.

## 8. 성공 기준 (MVP가 "됐다"고 말할 수 있는 조건)

1. 4시간짜리 녹음을 업로드하면, 설정한 전략으로 자동 커팅되어 세그먼트 wav들이 생성된다.
2. 각 세그먼트의 메타데이터가 자동 추출되어 DB에 저장되고 CSV로 내보내진다.
3. 파일명이 설정한 규칙대로 자동 생성된다.
4. 대시보드에서 총 개수·총 시간·용량·분포 통계가 보인다.
5. **차량음이 아닌 다른 도메인(예: 심음) Project를 설정만으로 새로 만들어** 위 과정을 반복할 수 있다. *(재사용성 검증 — 가장 중요)*

## 9. 가정과 제약 (Assumptions / Constraints)

- 단일 연구실, 로컬 환경에서 실행(MVP). 동시 사용자 소수.
- 백엔드는 Python 전용. 프론트는 React/Next.js.
- 개발은 Claude Code 기반 바이브 코딩. 따라서 **문서(PRD/아키텍처/CLAUDE.md)가 곧 개발 지침**이다.
- 오디오 처리는 로컬 CPU. 대용량(수 시간) 파일은 백그라운드 처리 필요(비동기 작업 고려 — 아키텍처 문서에서 다룸).

## 10. 용어집 (Glossary)

- **Project**: 하나의 연구 단위. 도메인 차이(커팅·라벨·명명 규칙)를 담는 설정의 집합.
- **Dataset**: Project 내부의 버전 있는 데이터 묶음.
- **Segment**: 커팅으로 생성된 개별 오디오 조각(1 레코드).
- **Label Schema**: 해당 Project가 요구하는 라벨 필드 정의(도메인마다 다름).
- **Storage Backend**: 파일을 저장하는 구현체 인터페이스(Local → Drive 교체 가능).
- **MCP**: 외부 서비스(Drive/Notion/GitHub)를 붙이는 연동 계층. 플러그인처럼 착탈.
