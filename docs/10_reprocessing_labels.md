# 10 — 재처리 시 라벨 보존 설계 (V2-4: 안전한 재커팅)

> **목적**: 재커팅이 기존 세그먼트의 라벨을 **조용히 잃어버리는 문제**를 해결한다.
> 실제 사고 사례에서 출발한 설계다. 승인되면 그대로 구현한다.
> 연결: 06_API.md §6(process 계약) · 05_database.md §3.3(Segment) → (이 문서)

- **버전**: v1.0 (설계 검토용)

---

## 1. 문제 정의 (실제 발생 사례, 2026-07-11~12)

dataset 5("aa" 프로젝트)에서 벌어진 일:

1. Job #8: `common_labels={'patient_id':'p03'}`로 커팅 → 30초 1조각 (threshold 문제)
2. 튜닝 과정에서 그 1조각을 삭제하고 파라미터를 바꿔 재커팅 (Job #12·#13)
3. 재커팅 요청에 common_labels를 다시 넣지 않음 → **새 33조각은 전부 라벨 없음.
   p03 라벨은 아무 경고 없이 소실.**

문제의 구조 (현재 동작의 두 가지 빈틈):

| 빈틈 | 현재 동작 | 결과 |
|---|---|---|
| **G1. 암묵적 라벨 손실** | 재커팅은 항상 "새 세그먼트 생성". 기존 라벨과의 연결이 없음 | 수작업 라벨(개별 PATCH 포함)이 재처리 한 번에 증발 |
| **G2. 중복 누적** | 같은 원본을 다시 process하면 기존 조각 위에 **새 조각이 누적** | 지금까지 사람이 기억해서 수동 삭제 (2회 발생) — CSV·통계도 이중 집계 |

연구 데이터에서 라벨은 사람의 시간이 들어간 **가장 비싼 자산**이다. 코드가 이를 지켜야 한다.

---

## 2. 설계 원칙

1. **명시적 동의 없는 파괴 금지**: 라벨이 있든 없든, 기존 세그먼트를 대체하는 재처리는
   호출자가 명시적으로 선언해야 한다 (기본값은 안전 쪽).
2. **승계는 기본, 포기는 선택**: 대체를 선언했다면 라벨 승계가 기본 동작.
   승계를 끄는 것이 예외적 선택이다.
3. **재현성**: 무엇을 대체했고 몇 개를 승계했는지 Job.params에 남는다
   (기존 재현성 체계 그대로 — Notion 로그에도 자동 반영됨).

---

## 3. API 계약 변경 (06_API.md §6.1 확장)

`ProcessRequest`에 필드 2개 추가:

```json
{
  "source_file_ids": [6],
  "params_override": null,
  "common_labels": {},
  "replace_existing": false,   // ★ 신규: 대상 원본의 기존 세그먼트를 대체
  "inherit_labels": true       // ★ 신규: 대체 시 라벨 승계 (기본 켜짐)
}
```

### 동작 규칙

| 상황 | 동작 |
|---|---|
| 대상 source_file에 기존 세그먼트 **없음** | 지금과 동일 — 커팅 진행 |
| 기존 세그먼트 **있음** + `replace_existing=false`(기본) | **409 CONFLICT** — "기존 세그먼트 N개(라벨 있는 것 M개)가 있습니다. replace_existing=true로 대체하거나, 보존이 필요하면 새 Dataset을 만들어 커팅하세요." |
| 기존 세그먼트 있음 + `replace_existing=true` | ① 기존 세그먼트 스냅샷(구간·라벨) → ② 기존 세그먼트 삭제(DB+Storage, Drive 미러 삭제 자동) → ③ 커팅 → ④ `inherit_labels=true`면 **겹침 매칭으로 라벨 승계** |

- **G2(중복 누적)는 409 가드가 해결**한다: 이제 같은 원본을 무심코 재커팅하면 누적 대신
  명시적 선택을 요구받는다. ※ 기존 동작(조용한 누적)의 의도적 변경 — §7 참조.
- `common_labels`와 승계가 겹치면: **승계된 라벨 위에 common_labels를 덮어쓴다**
  (이번 실행에서 명시한 값이 우선 — PATCH의 merge 의미론과 일관).

## 4. 라벨 승계 알고리즘 (worker)

```
스냅샷: 삭제 전 기존 세그먼트들의 (source_file_id, start_sec, end_sec, labels)

새 세그먼트 [s, e) 하나에 대해:
  1. 같은 source_file_id의 스냅샷 중 시간 겹침(overlap)이 있는 것을 찾는다
  2. 겹침 길이가 가장 큰 스냅샷의 labels를 승계
     (동률이면 새 조각의 중점을 포함하는 쪽)
  3. 겹침이 전혀 없으면 라벨 없음(빈 dict) — 옛 조각이 없던 구간이므로
  4. 승계 라벨을 label_schema로 재검증(type/enum) — 스키마가 그 사이 바뀌었을 수
     있으므로, 위반하는 키만 조용히 제외하고 로그를 남긴다
  5. common_labels가 있으면 그 위에 덮어쓰기 → is_labeled 재계산(빈 문자열 규칙 적용)
```

### 케이스 검증 (실사례 대입)

| 케이스 | 결과 |
|---|---|
| **파라미터 튜닝 재커팅** (30초 1조각 p03 → 33조각) | 옛 1조각이 새 33조각 전부와 겹침 → **33개 모두 p03 승계** ✅ (녹음 단위 라벨의 의도와 일치) |
| **같은 파라미터 재커팅** (경계 거의 동일) | 각 새 조각이 대응하는 옛 조각과 1:1 최대 겹침 → 개별 PATCH 예외 보정까지 그대로 승계 ✅ |
| **경계가 이동한 재커팅** | 최대 겹침 기준 근사 매칭 — 이벤트 단위 라벨은 경계 근처에서 어긋날 수 있음 (한계, §6) |

## 5. 구현 배치 (03 구조 준수)

| 파일 | 변경 |
|---|---|

| `tests/` | 가드 409 / 승계 1:1·1:N / 겹침 없음 / 스키마 변경 후 승계 / common_labels 우선순위 |

## 6. 한계 (문서화할 것)

- 승계는 **시간 겹침 기반 근사**다. 경계가 크게 이동하면 이벤트 단위 라벨(예: 특정
  비프의 방향)이 이웃 조각으로 옮겨갈 수 있다. 녹음 단위 라벨(patient_id 등)은 안전.
- 라벨의 "역사"는 남지 않는다(버저닝 비목표). 승계 전 상태가 필요하면 재처리 전
  CSV export가 스냅샷 역할을 한다 (Drive 미러 덕에 자동 백업됨).
- 완전한 보존이 필요한 실험은 **새 Dataset(v2)에 커팅**하는 것이 정답이다 —
  Dataset이 "버전 있는 묶음"인 이유(05 §3.2). 409 메시지에서도 이 경로를 안내한다.

---

## 7. 호환성 참고 (의도적 동작 변경)

기존: 같은 원본 재커팅 → 조용히 누적. 변경 후: **409로 차단**(replace_existing 요구).
프론트 ProcessingPanel은 409 메시지를 그대로 표시하므로 최소한의 UX는 즉시 확보된다.
"대체하고 재커팅" 확인 버튼(replace_existing=true 재요청)은 후속 프론트 작업으로 분리.
