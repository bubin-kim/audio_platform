# 14 — 푸시 알림 + 커팅 품질 검사 설계 (V2-6)

> **목적**: ① Job이 끝나거나 실패하면 연구원 휴대폰에 푸시 알림(ntfy),
> ② 커팅 결과가 **기대 조각 수와 다르면 감지해서 "재녹음 필요" 경고**
> (교수님 피드백 2026-07-16 — 30개가 안 나오면 현장에서 바로 알아야 함).
> 두 기능은 함께 구현한다: 품질 경고의 가치는 현장 푸시로 완성되므로.

- **버전**: v1.1 (품질 검사 통합, 사용자 승인 2026-07-16)
- **채널 결정 경위**: 카카오 알림톡은 사업자 인증+템플릿 심사 필요(보류).
  ntfy 채택 — 무료, 연구원은 가입 없이 앱 설치+토픽 구독만, 백엔드 연동은 HTTP POST 1회.
  Pushover(1인 $5)와 비교해 "사용 편의" 기준으로 선택 (사용자 위임 결정 2026-07-16).

## 1. 목표 / 비목표

### 목표
1. **커팅 완료** 알림: "✅ 커팅 완료 — {프로젝트} / {조각 수}개 ({총 길이}초)"
2. **커팅 실패** 알림: "❌ 커팅 실패 — {에러 요약}" (현장에서는 실패를 빨리 아는 게 더 중요)
3. **품질 검사 + 경고**: Project 설정 `expected_segments_per_source`(기대 조각 수/원본)와
   실제 조각 수를 원본 단위로 비교 — **부족·초과 모두 경고** (부족=녹음 누락/threshold 문제,
   초과=잡음 유입 가능성). 차단하지 않고 경고만; 화면 배지 + 푸시 + Job 기록.
4. P4 플러그인: `NTFY_TOPIC` 미설정이면 구독자 미등록 = 지금과 100% 동일 동작.
   `expected_segments_per_source` 미설정이면 품질 검사 안 함 (기존 프로젝트 무영향).

### 비목표 (후속)
- export 완료·업로드 알림 (필요해지면 구독자 함수 추가만)
- 사용자별 개인화 알림·읽음 확인·알림 이력 저장
- 카카오 알림톡 — 사업자 인증이 준비되면 같은 훅에 채널 추가

## 2. 동작 방식 (ntfy)

```
[worker] 커팅 Job 완료 ──▶ on_processing_done ──▶ hooks/ntfy.py 구독자
         커팅 Job 실패 ──▶ on_processing_failed(신설) ──▶      〃
                                                    │ (데몬 스레드, Notion 패턴)
                                                    ▼
                                    POST https://ntfy.sh/{NTFY_TOPIC}
                                    (제목·본문·우선순위·태그)
                                                    ▼
                                    연구원 휴대폰 ntfy 앱 푸시 (토픽 구독자 전원)
```

- **훅 신설 1개**: `on_processing_failed` — 지금은 실패 시 발화되는 훅이 없다.
  worker의 실패 처리 분기에 emit 한 줄 추가 (`dataset_id`, `job_id`, `error_msg`).
- 구독자는 notion.py와 동일 골격: 데몬 스레드(비차단) + 자체 DB 세션(상세 조회)
  + 실패는 로그만(본 흐름 보호).
- **토픽 이름이 곧 비밀번호다**: `audio-platform-{랜덤 12자}` 형태로 생성 —
  ntfy.sh 공개 서버에서 토픽을 아는 사람만 구독/발송 가능하므로 유출 시 교체.

### 알림 내용

| 이벤트 | 제목 | 본문 예 | 우선순위 |
|---|---|---|---|
| 완료(정상) | ✅ 커팅 완료 | `차량 경적음 테스트 / v1 — 30조각, 총 61.2초 (Job #12)` | 기본 |
| 완료+**품질 경고** | ⚠️ 조각 수 이상 | `29/30조각 (부족) — 재녹음 필요할 수 있음. horn_take3.wav (Job #14)` | **high** (진동) |
| 실패 | ❌ 커팅 실패 | `차량 경적음 테스트 / v1 — Job #13: 파일명 충돌: ...` (에러 앞 120자) | **high** (진동) |

### 품질 검사 동작 (원본 단위)

- Project 설정 `expected_segments_per_source`(nullable int, DB 컬럼 — 마이그레이션 1개).
  processing_service가 Job.params에 함께 기록(재현성).
- worker가 커팅 중 **원본별 조각 수**를 세고, Job 완료 시 `Job.params["quality_check"]`에 기록:
  ```json
  {"expected": 30, "sources": [{"source_file_id": 7, "filename": "take3.wav",
    "actual": 29, "status": "shortfall"}], "ok": false}
  ```
  status: `ok` | `shortfall`(부족) | `excess`(초과). 모든 원본이 ok면 `"ok": true`.
- 표시: 프론트 Job 완료 패널·Job 목록에 ⚠️ 배지 + 메시지, ntfy 경고 푸시, Notion 기록.
- 재녹음 워크플로우: 부족 원본을 삭제(DELETE /source-files) 후 재업로드, 또는
  `replace_existing` 재커팅 (V2-4 안전장치 그대로 적용).

## 3. 설정 (backend/.env → Railway env)

| 키 | 기본값 | 설명 |
|---|---|---|
| `NTFY_TOPIC` | `""` | 토픽 이름. **이것만 있으면 활성화** |
| `NTFY_SERVER` | `"https://ntfy.sh"` | 자체 호스팅 시 교체 가능 |
| `NTFY_TIMEOUT_SEC` | `5.0` | 발송 타임아웃 |

## 5. 운영 절차 (연구원 온보딩)

1. 휴대폰에 **ntfy 앱 설치** (App Store / Play Store, 무료)
2. 앱에서 `+` → 토픽 이름 입력(공지된 값) → 구독 완료 (가입 없음)
3. 끝 — 이후 커팅이 끝날 때마다 전원에게 푸시
