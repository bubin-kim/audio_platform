마일스톤 하나가 끝났을 때의 마무리 루틴. CLAUDE.md §10(완료의 정의)과
§9(커밋 하드 룰)를 순서대로 기계적으로 실행한다. 모든 경로는 저장소 루트 기준.

## 입력 (사용자에게 없으면 물어본다)

- **마일스톤 ID**: 예 `V2-3(D-M5)`, `V2-4(G-M1~M2)`. git log의 기존 접두사 형식과 일치시킨다.
- **한 줄 요약**: 커밋 제목에 들어갈 내용. 없으면 변경 diff를 보고 제안해 확인받는다.

## 단계 (순서 고정 — 실패하면 그 지점에서 멈추고 보고)

1. **상태 파악**: `git status --short` + `git log --oneline -3`.
   커밋 대상이 아닌 파일(스크래치, 로그)이 섞여 있으면 사용자에게 알린다.
2. **백엔드 테스트** (백엔드를 건드렸다면):
   ```bash
   cd backend && uv run pytest -q
   ```
   전부 통과해야 다음 단계로. 실패 시 커밋하지 않고 실패 내용을 보고한다.
3. **프론트 빌드** (프론트를 건드렸다면): dev 서버가 떠 있으면 안 된다
   (`lsof -i :3000 -sTCP:LISTEN` 확인, run-audio-platform 스킬 Gotchas의 .next 함정).
   ```bash
   cd frontend && npm run build
   ```
   빌드 후 `rm -rf frontend/.next` (사용자가 다음에 dev 서버를 깨끗하게 띄우도록).
4. **브라우저 검증** (UI를 건드렸다면): run-audio-platform 스킬로 확인.
   골든 패스에 영향 있으면 `node driver.mjs smoke`, 특정 화면이면 REPL로 그 화면만.
5. **코드 커밋** (저장소 루트에서):
   ```bash
   git add <해당 파일들>   # git add . 전에 1번에서 확인한 잡파일 제외
   git commit -m "<마일스톤ID>: <요약>

   - <주요 변경 bullet 2~5개>
   - 테스트 N개 통과"
   ```
   메시지는 **한국어**(CLAUDE.md §9 하드 룰). 커밋 후 해시를 확보한다.
6. **CLAUDE.md §11 갱신**: 방금 커밋 해시로 완료 항목 추가/진행 중 항목 갱신,
   "최신 마일스톤 커밋" 라인 갱신. 그리고:
   ```bash
   git add CLAUDE.md && git commit -m "docs: 진행 상황 갱신 — <마일스톤ID> 완료 (<해시>)"
   ```
   (코드 커밋 → 해시 확보 → 문서 커밋의 2-커밋 패턴. 예: `8496ebd` → `946e45f`)
7. **보고**: 두 커밋 해시, 테스트/빌드/브라우저 검증 결과, §11 반영 내용을 요약한다.

## 주의

- 커밋 메시지에 마일스톤 접두사·한국어는 **하드 룰** — 영어 메시지 금지.
- 테스트 실패·빌드 실패 상태로는 절대 커밋하지 않는다. 사용자가 명시적으로
  "그래도 커밋해"라고 하면 WIP임을 메시지에 표기한다.
- API·데이터 모델 계약이 바뀐 마일스톤이면 docs/06(및 해당 설계 문서) 갱신이
  커밋에 포함됐는지 5번 전에 확인한다 (CLAUDE.md §10-4).
